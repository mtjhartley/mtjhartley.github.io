HartPR

Purpose

Created to alleviate ranking debates and provide an archive for Washington Super Smash Bros. Melee tournaments, as well as an objective ranking of each player's skill in the region.

Technical Summary

HartPR was written in Python with the Flask framework and a PostgreSQL database. HartPR implements the Microsoft TrueSkill API as well as the APIs from two tournament bracketing websites, smash.gg and Challonge.

Personal Summary

For as long as I'd been involved in the Washington Melee community, players had been subjectively arguing about their skill level relative to their peers. I was inspired by GarPR, a webapp written by a YouTube engineer in NorCal and wanted to mirror its functionality. However, I had no prior experience in web development and was barely a few months into programming. Throughout the development of this project with the instruction of my friend Mitch Dzugan, I learned the fundamentals of web development and created HartPR.

My Process

1. Learn how to query the tournament bracket APIs and use their information as inputs into the Microsoft TrueSkill library to create meaningful data.

2. Learn SQL and store this data in a relational SQL database.

3. Render this data in an intuitive way on Flask-powered dynamically created HTML templates.

4. Host this data with Heroku and show it off to the WA Melee community.

Best Experiences

Setting and reaching goals. As I was new to web development, everyday working on the project presented a new challenge to me. Everyday I had a goals to accomplish, EX: (Learning to navigate a JSON response for the tournament data I wanted, storing the data in dictionaries and using the dictionaries for TrueSkill calculation, and saving the new TrueSkill results to the database).

This goal oriented programming kept me eager to learn and excited to accomplish new things, knowing I was ultimately working towards the goal of deploying a website that people in the community would use to track their ranks and tournament histories.

With every goal accomplished, I felt both a sense of satisfaction but also an eagerness to immediately start work on the next objective so that I could see the final product.

HartTV

Purpose

To create an app that lets you track what TV shows you've watched and record what you think of them, with a focus on community interaction.

Technical Summary

HartTV was written in Python with the Django framework, adhering to Django's MTV standard for development, with a MySQL database. HartTV implements the TVMaze API to pull information about Television shows and episodes. Standard CRUD features were implemented for the user community, shows, and reviews.

Personal Summary

This project was conceived out of the frustration of having to remember what TV episodes I had watched in years past that I wanted to recommend to my friends. I wanted to mirror the functionality of popular website MyAnimeList, which allows a user to track shows, discuss new episodes and view other users' profiles.

Development lasted one week during the end of my Python stack at Coding Dojo. During this week, I worked alone with occasional guidance from the Coding Dojo instructors. Going into this project after my experience with HartPR, I was much more confident in my design process and overall workflow. After designing the models, templates, and views, I outlined the list of features I wanted to accomplish and approached implementing them in logical order, aiming to produce the minimum viable product first. I proceeded to write the project within the allotted timeframe with efficient code and confidence.

StatsGG

Purpose

To visualize and interpret interesting data taken from Super Smash Bros. Melee tournaments recorded from a novel memory card hack created by Fizzi from smash.gg.

Technical Summary

StatsGG was written with MEAN (MongoDB, Express.js, AngularJS, Node.js) technologies to be an interactive website designed for data visualization.

StatsGG implements the smash.gg API, the YouTube IFrame API, and the ng2-chartsjs library to pull data, render videos with interactive timestamps, and dynamically generate graphs.

Personal Summary

StatsGG was created during project week at the end of my MEAN stack at Coding Dojo. I wanted to explore an area in programming I hadn't thus far in programming; data visualization and analysis. The main features I wanted to implement were a win probability graph, inspired by fangraphs.com, and the ability to dynamically generate timestamps for the YouTube video based on the frame starts of the combos from the smash.gg data. Accomplishing the win probability graph required revisiting linear regression and some college math.

I enjoyed writing code in Angular, as what I created was interactive and representative of the modern web. Throughout this project I gained a greater understanding of how JavaScript works in the browser, and with this understanding, was able to successfully merge Angular TypeScript with Vanilla JavaScript to produce the end product.

Codenames

Purpose

Created as an online version of the popular board game Codenames to have a fun family-friendly game to play on the weekends.

Technical Summary

The back-end for codenames was written in C# with the ASP.NET Core MVC framework and uses Entity Framework as the ORM to communicate with our MySQL database. The front-end was written in Angular4 with JavaScript, and is used to render different HTML elements for the codemasters and players.

Personal Summary

Codenames was developed in tandem with another student from Coding Dojo, Mike Bergstrom. We decided to team up on the project as neither of us had any prior meaningful experience pair programming with another person with similar skillsets and knowledge.

The most difficult feature to implement was the communication of the game state between the codemaster and the players. After discovering that the WebSockets and SignalR package was not compatible with our version of ASP.NET Core MVC, we decided to use our MySQL database to store all of the game state, and have both codemaster and players query the database intermittently depending on the phase of the game. Working out the logic of this process and the tradeoffs of using these frequent DB queries over socket based communication was educational and rewarding.

Best Experiences

Dividing up the work was a new experience for both of us in programming. We decided that I would handle much of the back-end queries and logic, while Mike would handle the front-end Angular code and managing the state of the game.

As the processes for communicating between the front-end and the back-end were so closely linked, we needed to clearly layout the desired inputs and outputs for each API call and response. Furthermore, we needed to keep each other on track with Git commits and discuss how to handle merge conflicts. This need to constantly communicate and explain our thought processes resulted in us writing cleaner and better code.
